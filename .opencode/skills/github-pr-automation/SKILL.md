# GitHub PR Automation

> **Version:** 1.0
> **Scope:** Automate PR creation, label assignment, reviewer assignment, and PR management
> **Integration:** Works with `/ship`, `/review`, and GitHub CLI (`gh`)

## When to Activate

Task keywords: `PR`, `pull request`, `create PR`, `merge`, `reviewer`, `label`, `assign`, `github`, `gh pr`

## PR Creation Workflow

### Step 1: Pre-PR Checks

Before creating a PR:

1. **Branch validation:**
   - Current branch is not `main` or `master`
   - Branch has commits ahead of target branch
   - Working tree is clean

2. **Quality gates:**
   - All quality gates passed
   - No lint errors, type errors, or test failures
   - Security scan passed (if configured)

3. **PR metadata:**
   - Title: Conventional commit format (`feat:`, `fix:`, `chore:`, etc.)
   - Description: Summary of changes, linked issues, testing notes
   - Labels: Auto-assigned based on change type
   - Reviewers: Auto-assigned based on code ownership

### Step 2: PR Creation

```bash
# Create PR with conventional title and auto-generated description
gh pr create \
  --title "<type>: <description>" \
  --body "<auto-generated description>" \
  --label "<auto-assigned labels>" \
  --reviewer "<auto-assigned reviewers>" \
  --base main
```

### Step 3: Auto-Generated PR Description

```markdown
## Summary
<One-line summary of changes>

## Changes
- <Change 1>
- <Change 2>
- <Change 3>

## Testing
- [ ] Quality gates passed
- [ ] Manual testing completed
- [ ] No regressions detected

## Related Issues
Closes #<issue-number>

## Deployment
- [ ] Preview deployment verified
- [ ] Production deployment planned
```

### Step 4: Auto-Label Assignment

| Change Type | Labels |
|---|---|
| `feat:` | `feature`, `needs-review` |
| `fix:` | `bug`, `needs-review` |
| `chore:` | `maintenance`, `needs-review` |
| `refactor:` | `refactor`, `needs-review` |
| `docs:` | `documentation`, `needs-review` |
| `test:` | `testing`, `needs-review` |
| `security:` | `security`, `urgent`, `needs-review` |
| `perf:` | `performance`, `needs-review` |

### Step 5: Auto-Reviewer Assignment

Based on code ownership and file types:

| File Pattern | Default Reviewer |
|---|---|
| `*.tsx`, `*.css`, `components/` | Frontend owner |
| `*.ts`, `*.js`, `api/`, `server/` | Backend owner |
| `prisma/`, `drizzle/`, `schema/` | Database owner |
| `auth/`, `crypto/`, `security/` | Security owner |
| `tests/`, `*.test.ts` | QA owner |

### Step 6: PR Status Checks

After PR creation:

1. **Monitor CI/CD:**
   - Check GitHub Actions status
   - Wait for all checks to pass
   - Report any failures

2. **Review Status:**
   - Track review approvals
   - Request changes if needed
   - Resolve review comments

3. **Merge Readiness:**
   - All checks passed
   - Required approvals received
   - No merge conflicts
   - Branch is up to date with target

## PR Management Commands

### Check PR Status
```bash
gh pr status
gh pr view <pr-number>
```

### Update PR
```bash
gh pr edit <pr-number> --title "new title"
gh pr edit <pr-number> --add-label "label1,label2"
gh pr edit <pr-number> --add-reviewer "user1,user2"
```

### Merge PR
```bash
gh pr merge <pr-number> --squash --delete-branch
```

## Integration with /ship

The `/ship` command now uses this skill for:
- Auto-generating PR descriptions
- Auto-assigning labels based on change type
- Auto-assigning reviewers based on code ownership
- Monitoring CI/CD status
- Tracking merge readiness

## Do Not

- Create PRs from `main` or `master` branches
- Create PRs without passing quality gates
- Assign reviewers without checking their availability
- Merge PRs without required approvals
- Create PRs with merge conflicts
- Skip security review for security-sensitive changes
